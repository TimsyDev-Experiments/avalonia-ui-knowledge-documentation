# 021V — Custom Controls from Scratch: An In-Depth Companion

**Why this exists.** The original tutorial shows the mechanics of `SignaturePad` — pointer handling, `Render` override, measure/arrange. This companion explains why you choose `Control` over `TemplatedControl`, how the `DrawingContext` pipeline works, when to override measure vs arrange, and the performance characteristics of each approach.

**Read this alongside:** [021 — Custom Controls from Scratch](021-custom-controls-from-scratch.md)

---

## 1. Why `Control` instead of `TemplatedControl`

```csharp
public class SignaturePad : Control
```

### The tradeoff

| Concern | `Control` | `TemplatedControl` |
|---|---|---|
| Visual definition | `Render(DrawingContext)` override | XAML template + `ControlTheme` |
| Consumer customization | Subclass + override `Render` | Replace the entire template |
| Performance | Fewer objects per instance | More objects (visual tree nodes) |
| Complexity | Logic and drawing in one class | Logic in C#, appearance in XAML |
| Use case | Drawing surfaces, game-like UIs | Reusable app controls (buttons, sliders) |

`Control` is the right base when your element's appearance is **procedural** — generated by code at runtime based on data that does not map well to XAML composition. A signature pad, a waveform viewer, a particle system, or a charting surface are all good candidates.

### What `Control` does NOT give you

- No `Template` property — you cannot swap appearances via XAML themes.
- No `OnApplyTemplate` — there are no named parts to wire up.
- No automatic `ControlTheme` lookup.
- No `ContentPresenter` — if you need to host arbitrary content, you must manage it yourself.

If later you need both procedural drawing AND template support, you can still extend `TemplatedControl` and put your custom drawing inside the template via a dedicated drawing control.

## 2. The static constructor and `AffectsRender`

```csharp
static SignaturePad()
{
    AffectsRender<SignaturePad>(BackgroundProperty, StrokeColorProperty, StrokeThicknessProperty);
}
```

### What `AffectsRender` does

`AffectsRender<T>` registers property-change watchers that call `InvalidateVisual()` whenever the listed properties change. Without this, changing `StrokeColor` from `Black` to `Blue` would update the stored property value but **would not cause the control to redraw**. The screen would show the old color until something else triggers a re-render.

The mechanism:
1. `AffectsRender` subscribes to the property's `Changed` event using a class handler.
2. When any listed property changes on any instance of `T`, the handler calls `InvalidateVisual()` on that instance.
3. At the next composition frame, the framework calls `Render(DrawingContext)` again.

### Properties that should be registered

Any property whose value is read inside `Render`. The original lists `BackgroundProperty`, `StrokeColorProperty`, `StrokeThicknessProperty`. If you miss one, the visual will not update when that property changes.

### `AffectsMeasure` and `AffectsArrange`

For properties that affect layout (not just rendering), use `AffectsMeasure<T>` or `AffectsArrange<T>`. These call `InvalidateMeasure()` and `InvalidateArrange()` respectively, which trigger a new layout pass before the next render.

## 3. Pointer event handling and capture

```csharp
protected override void OnPointerPressed(PointerPressedEventArgs e)
{
    base.OnPointerPressed(e);
    _isDrawing = true;
    _points.Clear();
    _points.Add(e.GetPosition(this));
    e.Pointer.Capture(this);
    InvalidateVisual();
}
```

### `e.GetPosition(this)` — coordinate space

`GetPosition(this)` returns the pointer position **relative to the specified element's origin** (top-left of `Bounds`). If you pass a different element, coordinates are in that element's local space. Always pass `this` (the control) so your drawing coordinates stay in the control's coordinate system.

### `e.Pointer.Capture(this)` — why capture matters

Pointer capture redirects all subsequent pointer events (`PointerMoved`, `PointerReleased`) to the capturing element, even if the pointer moves outside the element's bounds. Without capture, if the user drags quickly outside the `SignaturePad`:
- `PointerMoved` events go to whatever element is under the cursor (possibly nothing).
- `PointerReleased` goes to that other element.
- The `SignaturePad` never learns that the drag ended, leaving `_isDrawing = true` permanently.

Capture ensures the entire drag gesture is delivered to your control. You release capture in `OnPointerReleased` by calling `e.Pointer.Capture(null)`.

### Clearing `_points` on press vs. on release

The original clears `_points` in `OnPointerPressed`. This means each press starts a fresh stroke. If you want continuous drawing (press-move-release draws one stroke, next press starts another), this is correct. If you want to add to the existing strokes, clear in a separate "Clear" method and append in `OnPointerPressed`.

### Touch, pen, and mouse

Avalonia's pointer API unifies touch, pen, and mouse into `PointerEventArgs`. Use `e.Pointer.Type` to differentiate if needed:

```csharp
if (e.Pointer.Type == PointerType.Touch) { /* different handling */ }
```

## 4. The `Render` pipeline

```csharp
public override void Render(DrawingContext context)
```

### When `Render` is called

1. The element is first arranged (layout pass).
2. The element is visible (`IsVisible = true`).
3. `InvalidateVisual()` was called (by `AffectsRender` or manually).
4. The compositor's render thread picks up the invalidation and calls `Render` during the next frame.

`Render` is **not** called immediately when you call `InvalidateVisual()`. It is queued for the next composition frame. This batches multiple invalidations into a single render pass.

### The `DrawingContext` lifecycle

The framework creates a `DrawingContext`, calls your `Render`, then disposes it. The `DrawingContext` wraps a platform-specific render target (Direct2D, Skia, etc.). Your drawing commands are recorded and replayed on the render thread.

### Drawing order

Drawing commands execute in order, later draws occlude earlier ones:
1. Draw the background (so it appears behind everything).
2. Draw the border (on top of background).
3. Draw the stroke (on top of everything).

### What NOT to do in `Render`

- **Do not allocate** `Pen` or `Brush` objects every frame in performance-sensitive paths. Cache them as fields or use `Brushes.Black` / `Pens.Gray` from the static `Brushes` / `Pens` classes.
- **Do not trigger layout** — calling `InvalidateMeasure()` inside `Render` causes infinite re-layout.
- **Do not throw exceptions** — an unhandled exception in `Render` destabilizes the render loop.

### Optimization: cache the stroke geometry

For a signature pad with hundreds of points, `DrawLine` per segment creates many draw calls. For production, use `StreamGeometry` to build a single `Geometry` object:

```csharp
var geometry = new StreamGeometry();
using (var ctx = geometry.Open())
{
    if (_points.Count > 1)
    {
        ctx.BeginFigure(_points[0], false);
        for (int i = 1; i < _points.Count; i++)
            ctx.LineTo(_points[i]);
        ctx.EndFigure(false);
    }
}
context.DrawGeometry(null, stroke, geometry);
```

This sends one draw call to the GPU instead of N draw calls.

## 5. Measure and arrange

```csharp
protected override Size MeasureOverride(Size availableSize)
{
    var desired = new Size(300, 150);

    if (availableSize.Width < desired.Width)
        desired = desired.WithWidth(availableSize.Width);

    if (availableSize.Height < desired.Height)
        desired = desired.WithHeight(availableSize.Height);

    return desired;
}
```

### The layout contracts

Every `Control` implements a two-pass layout:

**Measure pass:** The parent tells this element how much space is available (`availableSize`). The element measures its children (if any) and returns its `DesiredSize`. The parent uses these desires to compute final positions.

**Arrange pass:** The parent tells this element its final position and size (`finalSize`). The element arranges its children within that space.

### When MeasureOverride is NOT called

- If the element is `Collapsed` (not `Visible`). Use `IsVisible` instead of WPF's `Visibility`.
- If the parent's `Measure` is not called (a bug in a custom panel).
- If `InvalidateMeasure()` was not called after a layout-affecting change.

### What happens if you return `availableSize` unchanged

The original `ArrangeOverride` returns `finalSize` unchanged — the control accepts whatever size its parent gives it. This is correct for a signature pad that fills available space. If you want a fixed aspect ratio, compute the constrained size in `ArrangeOverride` and return it.

### `ClipToBounds = true`

Set in the constructor. This prevents the stroke from drawing outside the control's bounds when the pointer moves near the edge. Without it, `DrawLine` still renders pixel data outside `Bounds`, which can leak into adjacent layout slots.

## 6. `InvalidateVisual()` and batching

```csharp
_points.Add(e.GetPosition(this));
InvalidateVisual();
```

Calling `InvalidateVisual()` on every `PointerMoved` event is correct for a signature pad — each new point should appear immediately. However, the compositor may coalesce multiple invalidations that arrive within the same frame interval (typically 16ms at 60fps). The render thread calls `Render` only once per frame regardless of how many times you called `InvalidateVisual()`.

For very high-frequency updates (drawing with touch on a 120Hz display), consider throttling:

```csharp
private DateTime _lastRender = DateTime.MinValue;

protected override void OnPointerMoved(PointerEventArgs e)
{
    base.OnPointerMoved(e);
    if (_isDrawing)
    {
        _points.Add(e.GetPosition(this));
        if ((DateTime.UtcNow - _lastRender).TotalMilliseconds > 8)
        {
            InvalidateVisual();
            _lastRender = DateTime.UtcNow;
        }
    }
}
```

## 7. Making the control bindable

The `StrokeColor` and `StrokeThickness` properties are styled properties, so they can be bound from XAML:

```xml
<controls:SignaturePad StrokeColor="{Binding PenColor}"
                       StrokeThickness="{Binding PenWidth}" />
```

If you want two-way binding for the stroke data (to save/load signatures), expose the points list as a `StyledProperty`:

```csharp
public static readonly StyledProperty<IEnumerable<Point>?> StrokePointsProperty =
    AvaloniaProperty.Register<SignaturePad, IEnumerable<Point>?>(nameof(StrokePoints));

public IEnumerable<Point>? StrokePoints
{
    get => GetValue(StrokePointsProperty);
    set => SetValue(StrokePointsProperty, value);
}
```

When `StrokePoints` is set externally, clear `_points` and repopulate from the value. When the user draws, update the property. This makes the signature data part of the binding graph.
